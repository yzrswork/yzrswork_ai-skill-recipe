## 1. プロダクトコンセプト

### 1.1 何であるか / 何でないか

| FieldCAD である | FieldCAD ではない |
|---|---|
| 手描きスケッチの置き換え | Fusion 360 / AutoCAD の置き換え |
| 「どこに何があるか」を数分で把握する道具 | 製造グレードの精密図面作成ツール |
| 寸法を「入力」したら図面が「生成」される | 線を「描く」ツール |
| 現場・作業台の横で片手で使うメモアプリ | デスクでマウスを使う設計ソフト |
| 加工前の「合うか?入るか?」の即席検証 | 公差・応力・干渉の厳密解析 |

### 1.2 解決する課題

電子工作・DIY・現場施工では、以下のような情報が **紙のメモや頭の中** に散らばりがちである。

- フロントアクリルの外形寸法と取付穴位置
- トグルスイッチ・DCジャック・LEDホルダーの穴径と位置
- 「この穴は表から加工、この皿モミは裏から」という加工面の区別
- 筐体の深さ 35mm に対して、スペーサー10mm + PCB 15mm + アクリル 3mm を積むと残りは何mmか

これらを紙に描くと、①清書に時間がかかる、②修正すると汚くなる、③別視点(側面・断面)を描き直す必要がある、④加工面(表/裏)の区別が曖昧になる、⑤クリアランス計算を暗算でミスする——という問題が起きる。

**FieldCAD は「寸法の入力」だけをユーザーに求め、作図・多視点展開・断面・残り空間計算をすべて自動化する。**

### 1.3 コアバリュー(優先順位付き)

1. **速さ** — 測って、タップして、入力したら終わり。1穴あたり10秒。
2. **明瞭さ** — 誰が見ても「どこに何があるか」が一目で分かる図。
3. **加工面の明示** — 表加工/裏加工の混在を色と記号で誤解なく伝える。
4. **空間の見える化** — Z方向の積み上げと残りクリアランスを自動計算・色分け。
5. 精度は「mm単位で正しい」レベルで十分。CAD級の厳密さは追わない。

## 2. ターゲットユーザー

### 2.1 ペルソナ

**ペルソナA:電子工作メイカー(主ターゲット)**
- 30〜50代。自宅で自作シンセ・計測器・ガジェットを製作。
- アクリルパネル+タカチ系ケースにPCB・スイッチ・ジャックを組み込む。
- 課題: 穴位置を方眼紙にメモ→加工時に読み間違える。PCBが筐体に収まるか組んでみるまで不安。
- 参考例(添付写真): アクリルフロントパネル+深型フレームに RP2350 ボード、LCD、エンコーダ、トグルSW、DCジャック、LEDホルダー、オーディオジャックを実装。この「穴8個+部品6点+奥行きの積み上げ」こそ本アプリの典型データ。

**ペルソナB:電気工事士・設備技術者**
- 現場で分電盤・制御盤の機器配置や貫通穴位置を記録。
- 課題: 現場写真+手書きメモだと事務所に戻ってから位置関係が思い出せない。
- ニーズ: 片手・立ち姿勢・軍手でも入力できる大きなUI。オフライン動作必須。

**ペルソナC:DIYユーザー・木工家**
- 棚・作業台・スピーカーボックスなどを製作。板取りと穴あけ位置の記録。
- 課題: CADは学習コストが高すぎて挫折。紙は無くす。
- ニーズ: 学習ゼロで使える。ホームセンターの加工サービスに見せられる図。

### 2.2 非ターゲット

- 機械設計者の業務用途(公差・図面規格 JIS/ISO 準拠が必要な人)→ 対象外
- 3Dモデリングをしたい人 → 対象外(3Dは「積み上げ計算と簡易断面」までに限定)

## 3. ユースケース

### UC-1: フロントパネルの穴あけ図を作る(最頻・最重要)
1. 「新規プロジェクト」→ パーツ追加「フロントアクリル 200×150×3mm」
2. 穴を追加: 「トグルSW φ12 @ (30, 40) 表から」「DCジャック φ11 @ (170, 120) 表から」…
3. 正面図が自動生成される。寸法線付き。
4. そのまま加工台の横にiPhoneを置いて、図を見ながらポンチ→ドリル。

**成功基準: 測定開始から図完成まで5分以内。**

### UC-2: 筐体内の奥行きが足りるか確認する
1. 筐体パーツ「フレーム 内寸深さ35mm」を登録。
2. Zスタックに積む: フロントアクリル3mm → スペーサー10mm → PCB(部品高さ15mm)。
3. アプリが残り 7mm を自動計算し、**緑**(余裕あり)で表示。
4. LCDを高さ18mmのものに変更 → 残り4mm → **黄**。さらに変更で干渉 → **赤**+警告。

### UC-3: 表/裏加工の指示を間違えない
1. 各穴に「加工面: 表/裏」を設定。
2. 正面図では表加工=実線+青バッジ、裏加工=破線+橙バッジで区別表示。
3. 「裏面ビュー」に切り替えると左右反転(ミラー)した裏から見た図を自動生成。
   → 裏から加工するときに、そのまま見ながら作業できる。

### UC-4: 現場で採寸メモとして使う(電気工事)
1. 電波の無い機械室でアプリを開く(オフライン動作)。
2. 盤面パーツを作り、既設機器をコンポーネントとして概略配置。
3. 事務所に戻って画面を見れば位置関係が完全に再現されている。

### UC-5: 過去プロジェクトの再利用
1. 以前作った「DCジャック穴 φ11」「トグルSW φ12」をコンポーネント/穴プリセットから呼び出す。
2. よく使う部品は自分用ライブラリに貯まっていき、入力がどんどん速くなる。

## 5. 非機能要件

| 分類 | 要件 |
|---|---|
| **性能** | 初回ロード3秒以内(4G回線)。描画操作60fps目標(最低30fps)。パーツ20個・穴200個・部品50個で快適動作。 |
| **オフライン** | 初回ロード後は完全オフライン動作(現場に電波は無い前提)。PWA化後はホーム画面起動で即オフライン。 |
| **データ保全** | 自動保存。アプリクラッシュ・タブ破棄でもデータ消失ゼロ。Safari の7日ITP対策として PWA インストール誘導+JSONエクスポート常時提供。 |
| **対応環境** | iOS Safari 16+ を第一級サポート。Android Chrome / デスクトップは「動けば良い」レベル。 |
| **操作性** | タップ領域は最小44×44pt(Apple HIG準拠)。片手・親指操作で主要フローが完結。軍手を想定し誤タップ耐性(undo必須)。 |
| **単位** | mm固定(小数1桁まで)。インチ対応は将来。 |
| **アクセシビリティ** | 屋外視認性のためハイコントラスト。文字サイズはOS設定に追従。色分けは色覚多様性を考慮し「色+記号/線種」の二重符号化。 |
| **セキュリティ/プライバシー** | データは端末内のみに保存。サーバー送信なし(将来のクラウド同期はオプトイン)。 |
| **保守性** | 依存ライブラリ最小。単一HTMLファイル+静的ホスティングで配布可能な構成(本リポジトリの他アプリと同方針)。 |
| **サイズ** | アプリ本体 500KB 以下(gzip後)。 |

## 6. 画面遷移図

画面数は意図的に少なくする。**「一覧 → ワークスペース」の2階層+モーダル**が基本。

```
┌─────────────────┐
│ ① プロジェクト一覧    │←──── 初回起動 / アプリアイコン
│  (ホーム)           │
└───┬─────────────┘
    │ プロジェクトをタップ / 新規作成
    ▼
┌─────────────────────────────────────────────┐
│ ② ワークスペース(メイン画面・アプリの本体)             │
│                                             │
│   下部タブで4モードを切替(画面遷移ではなくモード切替)   │
│  ┌─────────┬─────────┬─────────┬─────────┐  │
│  │ ビュー    │ パーツ    │ 部品      │ 断面/Z   │  │
│  │ (作図)   │ (材料+穴) │(ｺﾝﾎﾟｰﾈﾝﾄ) │(ｸﾘｱﾗﾝｽ)  │  │
│  └─────────┴─────────┴─────────┴─────────┘  │
└──┬──────┬──────┬──────┬──────────────────────┘
   │      │      │      │
   │      │      │      └─▶ [モーダル] 出力/共有シート
   │      │      │            (PNG / SVG / PDF / JSON)
   │      │      └─▶ [ボトムシート] コンポーネント編集フォーム
   │      │            (名前 / W×H×D / X,Y,Z / 色)
   │      └─▶ [ボトムシート] パーツ編集フォーム
   │            └─▶ [ボトムシート] 穴編集フォーム
   │                  (タイプ / 寸法 / X,Y / 加工面)
   └─▶ [オーバーレイ] ビュー切替パレット
         (正面 / 背面 / 上 / 左 / 右 / 断面)

補足:
・「編集フォーム」はすべて画面下から出るボトムシート。
  背後の図が見えたまま入力でき、数値変更が即座に図へ反映される。
・設定(単位・しきい値・テーマ)は ①ホームの歯車アイコンから。
・戻る操作は iOS 標準のエッジスワイプ&左上「<」を常設。
```

**設計意図:** CAD的な「モード地獄」を避けるため、深い階層を作らない。どの状態でも「図が見えている」ことを最優先し、入力UIは図に覆い被さらないボトムシートに統一する。

## 8. データモデル設計

### 8.1 設計方針

- **座標系を2種類に分離する**(ここが本アプリ設計の要):
  1. **パーツローカル座標系(2D)** — 穴の位置はパーツの左下原点からの X,Y。パーツをアセンブリ内でどう置こうと穴データは不変。加工図はこの座標系だけで描ける。
  2. **アセンブリ座標系(3D)** — パーツとコンポーネントの配置は X,Y,Z + 向き。多視点図・断面・クリアランスはこちらから導出。
- 図(ビュー)は**保存しない**。常にデータから導出する(Single Source of Truth)。
- 全IDは生成時に付与される不変ID。名前変更しても参照が壊れない。
- スキーマに `schemaVersion` を持ち、将来のマイグレーションに備える。

### 8.2 エンティティ関連図

```
Project 1 ─── * Part ──── * Hole
   │                         (パーツローカル2D座標)
   │ 1
   ├──── * Component          (購入部品。加工対象外)
   │ 1
   ├──── * Placement          (Part/Component をアセンブリ3D空間に配置)
   │ 1
   └──── 1 ZStack             (深さ方向の積み上げ定義)

UserLibrary(プロジェクト横断)
   ├──── * HolePreset          (φ12トグルSW穴 など)
   └──── * ComponentTemplate   (LCD 30×24×18 など)
```

### 8.4 設計上の重要判断

| 判断 | 理由 |
|---|---|
| 穴は Part に内包(独立テーブルにしない) | 穴はパーツと生死を共にする。参照整合性の管理を減らし、パーツ複製=穴ごと複製が自然に成立。 |
| 回転は90°単位のみ(orientation + flipped) | 任意角回転を許すと投影計算・UI・理解コストが爆発する。板と箱の世界では90°で実用上十分。**これが「CADにしない」ための最重要制約。** |
| Zスタックは placements からの自動導出+手動項目の併用 | スペーサー・ナット等いちいちコンポーネント登録したくない物を「厚みだけの行」として挿入できる。速度優先の思想に合致。 |
| 図は保存せず毎回導出 | データと図の不整合が原理的に発生しない。エクスポート形式追加も容易。 |
| ライブラリはプロジェクト外(UserLibrary) | プリセットの育成がプロジェクトを跨いで効く。エクスポートJSONには使用分のみ埋め込み、他端末でも開ける。 |

## 9. 推奨アーキテクチャ

### 9.1 全体構成

```
┌─────────────────────────────────────────────┐
│  単一HTMLファイル(fieldcad/index.html)         │
│  ※本リポジトリの他アプリ(yzrsynth等)と同じ配布形態  │
│                                             │
│  ┌───────────┐  ┌───────────┐  ┌──────────┐ │
│  │ UI層       │  │ 導出層     │  │ データ層   │ │
│  │ ボトムシート │◀─│ ビュー生成器 │◀─│ Store    │ │
│  │ タブ/フォーム│  │ 投影/断面   │  │ (状態)    │ │
│  │ SVG表示    │─▶│ ｸﾘｱﾗﾝｽ計算 │─▶│ undo履歴  │ │
│  └───────────┘  └───────────┘  └────┬─────┘ │

│                              ┌──────▼─────┐ │

│                              │ (+JSON入出力)│ │
│                              └────────────┘ │
└─────────────────────────────────────────────┘
         将来: manifest.json + Service Worker で PWA 化
```

### 9.2 技術選定と理由

| 領域 | 選定 | 理由 |
|---|---|---|
| フレームワーク | **Vanilla JS(フレームワーク無し)+ 単一HTML** | 本リポジトリの実績ある方針と一致。ビルド不要・静的ホスティングで完結・500KB制約に収まる。UIの複雑度はタブ4枚+シート数枚であり、フレームワークの恩恵よりロード速度と保守の単純さが勝る。 |
| 描画 | **SVG**(Canvasではなく) | ①ベクターなのでピンチズームで劣化しない ②要素ごとにタップ判定が自然に付く(穴=`<circle>`にイベント)③SVGエクスポートが実質無料で手に入る ④寸法線・テキストの品質が高い。パフォーマンスは穴200個規模なら全く問題ない。 |
| 状態管理 | 単方向データフロー(Store → render) | 「データから図を毎回導出」の方針と一致。状態変更は必ずアクション経由とし、その履歴が undo/redo に直結。 |
| 永続化 | **IndexedDB**(localStorage ではなく) | プロジェクトが増えると localStorage の5MB制限・同期APIのブロッキングが問題化。IndexedDB は非同期・大容量・構造化データ向き。 |
| データ保全 | 自動保存(debounce 500ms)+ JSONファイル書き出し | Safari ITP による7日データ削除リスクへの多重防御(§12参照)。 |
| PWA | 第2フェーズで manifest + Service Worker 追加 | リポジトリに既存の manifest.json / sw.js の知見あり。MVP はまずブラウザで価値検証。 |
| 幾何計算 | 自前実装(ライブラリ不使用) | 必要なのは AABB(軸平行境界箱)の投影・重なり判定・1次元区間の積み上げのみ。90°制約のおかげで三角関数すら不要。外部依存ゼロで十分書ける。 |
| 図面生成 | ビュー生成器 = 純関数 `project(data, viewType) → SVG要素記述` | 正面/背面/側面/断面はすべて「3D AABB集合を平面に投影する」同一処理のパラメータ違い。背面・左側面はX軸反転で得る(ミラー則)。 |
| PDF出力 | v1.0 で検討。SVG→印刷CSS(`@media print`)を第一候補、必要なら軽量PDFライブラリ | Safari の共有シート+「PDFとして保存」を活かせば依存追加なしで実現できる可能性が高い。 |
| DXF出力 | 将来。DXFはテキスト形式なので自前生成可能(R12形式、LINE/CIRCLE/ARCのみ) | レーザー入稿用途は輪郭+穴だけで足りる。 |

### 9.3 ビュー導出パイプライン(概念)

```
[データ] parts + placements + components
   │
   ├─ ①ワールド化: 各要素の AABB(x,y,z,w,h,d)をアセンブリ座標に展開
   │
   ├─ ②投影: viewType に応じて2軸を選び矩形/円に落とす
   │     正面図: (x, y)  背面図: (x, y)をX反転  上面図: (x, z) …
   │
   ├─ ③奥行きソート: 視線方向の座標で描画順を決定(奥→手前)
   │
   ├─ ④装飾: 加工面スタイル適用(表=実線/裏=破線+バッジ)、
   │     寸法線自動配置、レイヤー表示フィルタ
   │
   └─ ⑤SVG出力(画面表示・SVG/PNG/PDFエクスポートで共用)
```

### 9.4 モジュール分割(単一ファイル内の論理分割)

1. `store` — 状態・アクション・undo履歴・自動保存
2. `geometry` — AABB投影・区間積算・重なり判定(純関数群、単体テスト対象)
3. `views` — ビュー生成器(純関数)
4. `ui` — タブ・ボトムシート・フォーム・ジェスチャ
5. `io` — IndexedDB・JSON入出力・画像/SVGエクスポート
6. `presets` — 標準穴プリセット・コンポーネント雛形データ

## 11. UI/UX設計思想

### 11.1 五つの原則

1. **「描く」を禁止し「入力」に徹する**
   フリーハンド描画・ドラッグでの図形作成は提供しない。曖昧な操作は曖昧な図を生む。数値入力だけが正確な図を最速で生む。ドラッグは「閲覧(パン・ズーム)」と「微調整(選択済み要素の0.5mmスナップ移動)」に限定する。

2. **常に図が見えている**
   フルスクリーンのフォーム画面を作らない。入力は必ずボトムシートで、背後の図がリアルタイムに更新される。「入力→反映」のフィードバックループが0秒であることが、CADに勝る体験の核。

3. **1穴10秒**
   最頻操作(穴追加)の所要タップ数を最小化する: [+穴] → プリセット選択 → X入力 → Y入力 → 完了(タップ5回+数字入力)。「複製して次へ」で連続入力時はさらに短縮。

4. **メモ帳の気軽さ、図面の信頼性**
   見た目はカジュアル(角丸・大きな文字・絵文字アイコン可)だが、出力される図は寸法線・注記が整った「見せられる図」。この落差が価値になる。

5. **間違いは防ぐより戻せるように**
   現場では誤タップが必ず起きる。確認ダイアログを増やすのではなく、全操作をundo可能にし、削除はスナックバーで即時復元できるようにする。

### 11.2 具体的なUIルール

- タップ領域 44pt 以上。主要ボタンは画面下半分。
- 数値入力は `inputmode="decimal"`+ステッパー(±0.5、長押し連続)。キーボードを出さずに微調整可能。
- 表加工=**実線+青**、裏加工=**破線+橙**+「裏」バッジ。色だけに依存しない二重符号化(色覚対応・白黒印刷対応)。
- クリアランス: 緑✅ / 黄⚠️ / 赤⛔ のアイコン併記(同上の理由)。
- ダークモード対応(OS追従)。屋外では明るいテーマ、暗所作業では暗いテーマが効く。
- 用語はCAD用語を避ける:「正投影図」ではなく「正面から見た図」。「フィレット」ではなく「角の丸み」。

### 11.3 オンボーディング

- チュートリアル画面は作らない。代わりに初回起動時、**サンプルプロジェクト「シンセ筐体(例)」**(添付写真相当のデータ)がプリインストールされており、触って構造を理解できる。
- 空状態には常に次の一手を表示(「まだパーツがありません → [+ パーツを追加]」)。

## 12. 想定される問題と対策

| # | 問題 | リスク | 対策 |
|---|---|---|---|
| 1 | **Safari ITP による7日間未使用でのストレージ削除** | 致命的(データ消失) | 多重防御: ①PWAインストールを促す(ホーム画面追加でITP対象外)②ワークスペース退出時にJSONバックアップを促すバナー ③共有シートでファイルアプリ/iCloudへワンタップ保存 ④「最終バックアップ日時」をホームに常時表示。 |
| 2 | **3D的な位置関係(特にZ)の入力が直感的でない** | 中(学習コスト) | Z入力を「数値」ではなく**Zスタック(積み上げリスト)への挿入**として提供。「アクリルの後ろにスペーサーを積む」という物理的メンタルモデルに合わせる。数値直接入力は上級者向けに残す。 |
| 3 | **裏面加工のミラー混乱**(裏から見るとXが反転し、現場で穴位置を間違える) | 高(実害: 材料をダメにする) | ①裏面ビューは常にミラー済みを表示し「裏から見た図(左右反転済み)」と明記 ②穴座標表には「表基準X / 裏基準X」を併記 ③裏加工穴には図中バッジで注意喚起。 |
| 4 | 小画面で寸法線が密集して読めない | 中 | ①ズーム率に応じた寸法線の間引き表示 ②「選択した穴の寸法だけ表示」モード ③寸法は図中でなく穴一覧テーブルでも確認可能にする(図と表の二本立て)。 |
| 5 | スコープクリープで「ミニCAD化」してしまう | 高(コンセプト死) | 設計制約として明文化: 任意角回転なし・スケッチ機能なし・3Dビューなし・拘束(コンストレイント)なし。新機能は「入力が速くなるか?図が分かりやすくなるか?」の2問に Yes でなければ却下。 |
| 6 | ピンチズームとスクロール・要素選択のジェスチャ競合 | 中 | 描画領域では `touch-action: none` で自前ハンドリングに統一。1本指タップ=選択、1本指ドラッグ=パン、2本指=ズーム、要素移動は「選択してから表示されるハンドル操作のみ」に限定し誤爆を防ぐ。 |
| 7 | 自動配置した寸法線同士の重なり | 中 | MVPでは単純な段積み(パーツ外側に1段目・2段目…)アルゴリズムで妥協。完璧な自動レイアウトは追わず、v1.0で「寸法線の手動オフセット」を追加。 |
| 8 | IndexedDB書き込み失敗・容量制限(プライベートブラウズ等) | 低〜中 | 起動時に書込テストを行い、失敗時は「保存できない環境」バナー+メモリ動作+JSON書き出し誘導。 |
| 9 | 円形パーツ・異形状(L字等)への要望 | 中 | MVPは矩形+円のみと明言。異形は「矩形の組み合わせ(複数パーツ)」で近似運用してもらい、需要が実証されたら多角形対応を検討。 |
| 10 | 単位間違い(cm/mm混同)による入力ミス | 低 | 全入力欄に単位「mm」を常時表示。異常値(例: パーツより大きい穴座標)には保存は許しつつ警告アイコンを表示(入力を止めない=速度優先)。 |

## 13. プロダクトを大きく伸ばす追加機能アイデア

優先度 ★=高い(v1.0〜早期に検討価値あり)

1. **★ 穴座標の「加工チェックリスト」モード**
   穴一覧を大きなチェックボックス付きリストで表示し、加工済み穴にチェックを入れていく。ドリル作業中の「どこまで開けたっけ?」を解消。図側でもチェック済み穴がグレーアウト。**加工現場での実用性が跳ね上がる、実装コストの低い killer feature。**

2. **★ ドリル径サマリー**
   プロジェクト内の全穴を径ごとに集計(「φ3.2×8個、φ11×1、φ12×1」)。作業前に必要なドリルビットが一覧でき、ビット交換回数を最小化する加工順の提案にも発展可能。

3. **★ 材料コストメモと板取り(ネスティング)簡易版**
   パーツに単価メモを持たせ合計表示。さらに「600×450のアクリル板からこのパーツ群が取れるか」の手動板取りシミュレーション。DIY層に強く刺さる。

4. **測った値をその場で入れる「クイック採寸モード」**
   図を経由せず、名前と数値だけを高速連続入力できる電卓風UI。後からワークスペースで整理する。「とにかく今すぐメモしたい」現場ニーズに応える。

5. **2点基準の写真オーバーレイ(将来機能F-81の具体化)**
   実物の写真上で既知の2点間距離を指定→スケール較正→図を半透明重畳。既存機器の穴位置を「写真からおおよそ拾う」ことができ、採寸の手間を半減。

6. **ARクイックルック連携(iOS特化)**
   ZスタックデータからUSDZを生成し、iOS標準のARビューで実寸表示。「机の上に実寸の完成品が浮かぶ」体験はiPhoneファースト戦略と相性が最良。

7. **閲覧専用共有リンク**
   プロジェクトJSONをURLフラグメント(#以降)に圧縮エンコードして共有。サーバー不要のまま「加工をお願いする相手に図を送る」が実現。ホームセンター加工サービスや共同製作で効く。

8. **ねじ・スペーサー計算アシスタント**
   「板厚3+基板1.6+スペーサー10 → 必要ねじ長は?」を自動計算し、M3×15などの規格品を提案。Zスタックデータを持つ本アプリだからこそ出せる付加価値。

9. **テンプレートギャラリー**
   「1590Bサイズエフェクターケース」「タカチSW-125」「ユーロラック3Uパネル」等の定番筐体をワンタップで開始点にできる。メイカーコミュニティでの拡散起点になる。

10. **変更履歴のタイムライン**
    undo履歴を日付付きで保持し「昨日の状態に戻す」を可能に。現場で「やっぱり前の配置が良かった」に応える。
