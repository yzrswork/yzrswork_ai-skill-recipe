# 作業ログ 2026-06-12

## 概要
Xでブックマークしたツイートを自動的にGitHub上のObsidian VaultへMarkdown保存するユーザースクリプト
「X Bookmark Logger」を新規作成。

- ブランチ: `claude/x-activity-logger-dkjadb`
- 成果物: `x-bookmark-logger.user.js` / `x-bookmark-uploader.user.js` / `x-bookmark-logger.html`

## 方式の検討

X公式APIは2026年2月以降、新規開発者に無料枠がなく従量課金のみ（自分のデータ読み取りは約$0.001/件 +
開発者登録・支払い設定が必要）。そのため無料で使える**GraphQL傍受方式**
（[prinsss/twitter-web-exporter](https://github.com/prinsss/twitter-web-exporter) と同系の手法）を採用した。

- 記録対象: ブックマークのみ（ユーザー要望。いいねは対象外）
- 保存先: ユーザーのGitHub Obsidian Vaultリポジトリ（contents API + fine-grained PAT、設定UIから指定）
- xarchive方式（queryIdを能動的に叩く）はqueryIdローテーションに弱いため不採用。
  X本体が発行した通信に相乗りする傍受方式はローテーションの影響を受けない。

## アーキテクチャ

### x-bookmark-logger.user.js（本体）
- `@run-at document-start` で `XMLHttpRequest.prototype.open/send` + `fetch` をパッチ
- **リアルタイム経路**: `CreateBookmark` ミューテーションを検知 → tweet_id はリクエストボディの
  `variables.tweet_id` から取得（レスポンスは `{"tweet_bookmark_put":"Done"}` のみで本文を含まないため）、
  本文・作者は全GraphQLレスポンスを deep walk して作る**ツイートキャッシュ**（直近500件、localStorage永続）から補完
- **差分同期経路**: `Bookmarks` タイムライン（`/i/bookmarks` のスクロール）の entries から
  `entryId: tweet-` のみ抽出（deep walkだと引用元まで拾うため instructions パスで判定）。
  iPhoneのXアプリでブックマークした分はこの経路で取り込む
- **重複防止は二段構え**: localStorageの記録済みIDセット + ファイル名を「投稿日_screenname_ID.md」で
  決定的に生成し、GitHub側の422 (file exists) を「記録済み」シグナルとして扱う
  （ファイル名の日付に bookmarked_at ではなく posted_at を使うのは、端末間でズレない決定的な値にするため）
- **コミット層**: localStorageキュー → GM_xmlhttpRequest で直列PUT（1.1秒間隔、409はバックオフ、
  401/403は停止して警告、5xx/オフラインはキューに残して自動再送）。タブ間/スクリプト間はlocalStorageロックで排他
- **設定UI**: x.com左下のフローティング🔖ボタン → パネル（owner/repo/branch/folder/PAT/接続テスト/
  未送信バッジ/手動フラッシュ/索引クリア）
- 抽出は新旧ペイロード両対応: screen_name は `user.legacy` と `user.core` 両方フォールバック、
  長文は note_tweet 優先、t.co展開、HTMLエンティティ復号、動画は最高ビットレートmp4、引用はObsidian callout

### x-bookmark-uploader.user.js（iOSコンパニオン）
iOSのUserscriptsアプリは「GM APIはcontentワールドのみ / XHRパッチはpageワールドのみ / x.comのCSPで
素のfetchはapi.github.comに届かない」という三すくみのため、本体（page側・傍受担当）と
アップローダー（content側・送信担当）の2本構成にし、localStorageキュー + CustomEventで橋渡しする。
本体がGM送信可能な環境（PC）では `xbm.cap_gm` フラグでアップローダーは何もしない。

## 検証

Playwright + スタブサーバ（偽GraphQLエンドポイント + 偽GM_xmlhttpRequest）で結合テストを実施し全件PASS:
1. TweetDetailキャッシュ → CreateBookmark → 正しいファイル名/フロントマター/引用calloutでPUTされる
2. Bookmarksタイムライン → 2件差分コミット（Tombstoneはスキップ、新旧screen_name形式両対応、
   メディア抽出、t.co展開、HTMLエンティティ復号）
3. 同じタイムラインを再度流しても再コミットされない（記録済み索引）
4. fetch経由の傍受 + 422レスポンスを記録済みとして処理（キューから除去・索引登録）
5. 設定UI（フローティングボタン・パネル開閉・設定値反映）

実環境（本物のx.com / 実PAT）での動作確認は未実施。GraphQLレスポンス構造は
twitter-web-exporterのソースから確認した構造に基づくが、X側の仕様変更リスクがあるため、
初回利用時は `__XBM_DEBUG.state()` でキュー・キャッシュの様子を確認すること。

## セキュリティ上の注意（ガイドHTMLにも記載）
- PATはx.comのlocalStorageに置くため、fine-grained PATで「Vaultリポジトリ1つ・Contents権限のみ・短期限」に限定
- 非公式手法のためToS的にはグレー（読み取りのみ・自分のデータのみでリスク小と判断）
